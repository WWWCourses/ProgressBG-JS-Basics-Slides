What are Git and GitHub

What are Git and GitHub

Git is a distributed Version Control System (VCS) that tracks changes in source code during software development. It allows multiple developers to work on a project simultaneously without overwriting each other's changes.
GitHub is a web-based platform that hosts Git repositories. It provides additional tools and features to facilitate collaboration, such as pull requests, issue tracking, code reviews, and project management. GitHub also enables developers to share code publicly or privately, contributing to open-source projects or working on private repositories.
There are many GitHub alternatives, like GitLab and Bitbucket.
Git vs Github: while Git is the underlying technology for version control, GitHub adds a layer of collaboration and cloud storage.

What is a VCS and how it makes life easier?

What is a VCS and how it makes life easier?

VCSVersionControlSystem
a system that records changes in a project (files and directories) over time so that you can recall specific versions later
You can version control any files, all kind of work you do: writing, programming, images.
But VCS are extremely helpful in software development

VCS Advantages in Software Development

Track Changes Over Time: VCS allows developers to keep a history of changes made to the codebase. This feature is invaluable for understanding how a project has evolved and for analyzing the impact of specific changes.
Facilitate Collaboration: In modern software development, teams often work on the same projects simultaneously. VCS supports multiple developers working on the same codebase without overwriting each other's changes, merging contributions seamlessly.
Error Recovery: When errors are introduced into the code, VCS makes it easy to revert to earlier, error-free versions. This capability significantly reduces the risk associated with new deployments.
Branching and Merging: VCS enables developers to create branches, allowing them to work on new features or fixes independently from the main codebase. Once ready, these changes can be merged back into the main project, facilitating continuous integration and delivery.
Track Who Made Changes: VCS provides transparency over who made specific changes, adding an accountability layer to the development process. This feature is crucial for coordinating team efforts and resolving issues.

Types of VSC

Types of VSC

Local VCS

Local version control systems store tracking information on the local computer only.
While it's a simple method, it's not commonly used by companies due to the lack of collaboration features and increased risk of data loss.

Centralized VCS

Centralized version control systems, like Subversion (SVN), use a single server repository and client workstations that check out files from that repository.
Centralized VCSs simplify collaboration but introduce a single point of failure that can compromise the entire project's history if not properly backed up.

Decentralized (Distributed) VCS

Distributed version control systems, like Git, allow each user to have a full copy of the entire repository history, enhancing collaboration and security.
This model supports distributed development, allows for multiple backup points, and facilitates complex merging and branching strategies.

Distributed VCS Concepts

Distributed VCS Concepts

Overview

Working Copy (Working Directory):
The actual files a developer works on. Changes made here can be saved (committed) to their local repository.
Repository
It's a place where all the project's files live and where you can see the history of what's been done, who did it, and when. Repositories can be local or remote
Local Repository:
Local repositories reside on the computer of each developer, allowing for individual work on the project's codebase. Each contributor has their own local repository with the full history of the project.
Remote Repository:
Remote repositories are hosted on the internet or a network, enabling multiple developers to push and pull changes, thus supporting collaborative work across teams.
Commit (Local):
Saving changes to the local repository. It's a snapshot of the working copy at a given time.
Developers must commit their work locally before sharing it with others.
Pull:
Bringing in changes from the server repository to the local repository to keep it up-to-date.
Push:
Transferring local changes (commits_ to the server repository. This is how changes are shared with other team members.

Setting Up Git

Setting Up Git

Installing Git

Visit the official Git website (https://git-scm.com/downloads ) to download the Git installer for your operating system.
Follow the installation wizard, which will guide you through the setup process, including selecting the right options for your environment.
To verify the installation, open your terminal (Command Prompt, PowerShell, or Terminal) and type:

                git -v

                # git version 2.43.0

Setting up user name and email

Before your first git commit, you have to provide your user name and email.
This information will be stamped in each of you commits and other git activities you make.
You are not obliged to put your GitHub account email, but keep in mind that GitHub relies on the email address associated with commits to attribute contributions to user profiles. If the email address in your Git configuration does not match the one associated with your GitHub account, your contributions will not be reflected on your GitHub profile activity and contribution graphs.

                #Global:  if you wish to setup your git identity on global level, you can use the --global option:
                git config --global user.email "your_mail@mail.com"
                git config --global user.name "User Name"

                # Local: per project - make sure you type next commands while you are in the root git folder
                git config user.email "your_mail@mail.com"
                git config user.name "User Name"
Check your configuration by typing:

                git config user.name
                git config user.email

Set VSCode as default git editor

If you want to configure Git to use VSCode as the default editor for commit messages and other interactive Git commands:

                git config --global core.editor "code --wait"

Saving changes in Git

The Three States

Understanding the three states in Git - working directory, staging area, and repository - is crucial for effectively managing and saving changes.
By comprehending these states, developers gain clarity on the status and progression of their code modifications
In Git, there are three main states that files can reside in within a Git repository:
Working Directory (Untracked files): This is the state where files exist in your project directory but have not yet been tracked by Git. Git is unaware of any changes made to these files, and they are considered untracked.
Staging Area (Staged files): After modifying files in the working directory, you can stage them for the next commit by adding them to the staging area using the git add command. Files in the staging area are ready to be committed, and Git is aware of the changes made to these files.
Repository (Committed files): When you commit changes in the staging area using the git commit command, Git creates a snapshot of the staged changes and permanently stores them in the repository. These committed changes are now part of the project's history and can be accessed and referenced in the future.

The Staging Area

Overview

The staging area, also known as the index, is a crucial concept in Git that serves as a middle ground between your working directory and the repository. It acts as a temporary storage area where changes are prepared before being committed to the repository.
Imagine the staging area as a preparation zone where you carefully select and organize items (changes) before packing them into a box (commit) for shipping (pushing to the repository).
Before committing changes to the repository, files must be added to the staging area. This allows you to selectively choose which changes to include in the next commit, providing finer control over the commit process.
The staging area allows for the separation of logical changes into smaller, more manageable commits.
Common Git commands related to the staging area include:
  • git add: Adds changes from the working directory to the staging area.
  • git reset: Removes changes from the staging area, but keeps them in the working directory.
  • git status: Displays the status of files in the working directory and staging area.

Add changes to stage (index): git add

To stage changes only in certain file/directory:

                git add path
To stage all changes in the working directory, including new, modified, and deleted files (including untracked files and directories):

                git add -A

                # or the shortcut:
                git add .

Remove files from the staging area: git reset

To remove files from the staging area in Git, you can use the "git reset" command with the "--" option followed by the filenames of the files you want to unstage.

                git reset HEAD -- <filename>
After running the "git reset" command, the specified files will be removed from the staging area, but their changes will still be present in your working directory.
Use this command cautiously. Make sure you understand the consequences before using it.

Commit changes: git commit

Git Commit Command

The "git commit" command is used to save snapshot of the staged changes and adds them to the repository's history.

                git commit -m "commit message"
The "-m" flag is used to specify a commit message, which provides a brief description of the changes made in the commit.
Commit Best Practices:
Write clear and concise commit messages that describe the purpose of the changes.
Commit small, focused changes rather than large, sweeping modifications.
Review changes before committing to ensure that only intended modifications are included.
Use present tense in commit messages and avoid unnecessary details.
Examples:

                "Fix typo in user registration form" #GOOD
                "Changed registration form"          #BAD

                "Add validation for email address in registration form"  #GOOD
                "Update registration form functionality"                 #BAD

Git Status and Repository History

git status

Overview

git status is a command in Git that displays the current state of the working directory and staging area. When you run "git status", Git provides information about which files have been modified, which files are staged for the next commit, and which files are untracked. Here's what "git status" typically shows:
Modified Files: Files that have been modified in the working directory but have not yet been staged for commit. Git informs you of changes that have been made since the last commit.
Staged Files: Files that have been modified and staged using "git add" for the next commit. These changes are ready to be included in the next commit.
Untracked Files: Files that are present in the working directory but have not been added to the Git repository. Git does not track changes to these files unless they are explicitly staged using "git add".
Branch Information: Git also displays information about the current branch and whether there are any commits ahead or behind the remote branch, if you're working with a remote repository.

Example


                git status

                # Output:
                On branch main
                Your branch is up to date with 'origin/main'.

                Changes not staged for commit:
                  (use "git add <file>..." to update what will be committed)
                  (use "git restore <file>..." to discard changes in working directory)
                        modified:   index.html

                Untracked files:
                  (use "git add <file>..." to include in what will be committed)
                        newfile.txt

                no changes added to commit (use "git add" and/or "git commit -a")
The status tell us that:
The branch is "main" and it's up to date with the remote branch.
There is one modified file ("index.html") that has changes not staged for commit.
There is one untracked file ("newfile.txt") that Git has not yet recognized.
Git provides suggestions on how to stage changes for commit ("git add") or discard changes ("git restore").

View Repository History

git log

The "git log" command is used to view the commit history of a Git repository.
When you run "git log", Git displays a chronological list of commits, starting with the most recent commit at the top.
The output of "git log" includes details such as the commit hash (unique identifier assigned to each commit in a Git repository), author, date and time of the commit, and the commit message.
There are various options you can use with "git log" to customize the output, such as "--oneline" for a concise view, "--graph" for a graphical representation of branches, and "--since" or "--until" to filter commits by date.
"git log" is a valuable tool for understanding the history of a project, tracking changes, and collaborating with team members. It provides insight into who made changes, when they were made, and the purpose of each commit.
Example:

                git log

                commit 3e2f0a7f8eaa1e6b9282cb108de41a62aefa1253 (HEAD -> main)
                Author: John Doe <johndoe@example.com>

the .gitignore file

the .gitignore file

Overview

The .gitignore file is used in Git to specify intentionally untracked files that Git should ignore. These files are typically generated as part of the build process or contain sensitive information and you do not want them to be in the repository.
Examples of files often included in .gitignore are compiled binaries, log files, temporary files, and configuration files containing sensitive information like API keys or passwords.
By creating a .gitignore file in the root directory of the repository and listing patterns for files or directories to ignore, Git will automatically disregard those files when tracking changes.
.gitignore uses simple pattern matching rules to specify which files or directories to ignore. Patterns can include wildcards such as "*" to match multiple files or directories, and "!" to negate patterns.

                # Ignore all .txt files in the repository
                *.txt

                # Ignore all files in the "logs" directory
                logs/*

                # Ignore all files with the .log extension, except "important.log"
                *.log
                !important.log

                # Ignore the "config" directory and its contents
                /config/

                # Ignore all files with the .tmp extension in any directory
                **/*.tmp

                # Exclude all files in the "docs" directory, including subdirectories
                /docs/

Introduction to Git Branches

Introduction to Git Branches

What are branches?

Imagine you're working on a group project, and you want to try out new ideas without affecting the main project. Git branches let you do just that.
Each branch in Git represents a separate line of development. Think of them as parallel versions of the project that you can work on simultaneously. This way, you can experiment, develop new features, or fix bugs in isolation from the main project (often referred to as the 'main' or 'master' branch).
Once you're satisfied with the changes on a branch, you can merge them back into the main project, ensuring that all improvements are incorporated seamlessly. This approach not only simplifies development but also minimizes the risk of disrupting the main codebase.

Creating and Switching Branches

To create a new branch, use the git branch command followed by the name of the new branch.
This creates a new branch but does not switch to it.

                git branch new-feature
To switch to an existing branch, use the git checkout command.
This command changes the current working directory to the specified branch.

                git checkout new-feature

Merging Branches

Merging is a fundamental Git operation that integrates changes from one branch into another.
Think of merging like combining the ingredients of two recipes into one. In Git, merging takes the changes from one branch (like a list of ingredients) and adds them to another, making your project up-to-date with all the changes.
How to Merge
1. Switch to the branch you want to add changes to. This is usually your main project branch, often called main.

                git checkout main
2. Bring in the changes from another branch, say feature, into your main branch.

                git merge feature
This tells Git: "Take everything from feature and add it to main."

Conflict Resolution

Merge conflicts occurs when changes in one branch clash with changes in another.
It's like when two recipes suggest different oven temperatures - you have to decide which one to follow.
Git will highlight conflicts during a merge, and you'll have to edit the files to fix them. After that, you tell Git you've resolved the conflicts by adding and committing the changes.

Best Practices for Branch Management

Regularly merge changes from the main branch into your feature branches.
This helps minimize merge conflicts and keeps the branch up to date.
Delete branches after their changes have been merged into the main branch.
This helps to keep the repository clean and manageable.

Conflict Resolution: A Simple Scenario

You and a teammate have made different changes to the same part of a file named index.html. Your work involved adding a new H2 element to index.html, while your teammate has modified the text of an existing H1 element in the same file.
You try to merge your teammate's branch feature into your current branch main:

            git checkout main
            git merge feature
and a conflict occurs. Here is the output

            # Auto-merging index.html
            # CONFLICT (content): Merge conflict in index.html
            # Automatic merge failed; fix conflicts and then commit the result.
To fix the conflict, you open index.html, and will see the conflicting changes marked by Git:

            <<<<<<< HEAD
            <h1>Original H1 Title</h1>
            <h2>Your New H2</h2>
            =======
            <h1>Teammate's Modified H1</h1>
            >>>>>>> feature
You decide to keep both the modified H1 and your new H2, thus editing the file to reflect both sets of changes.

                <h1>Teammate's Modified H1</h1>
                <h2>Your New H2</h2>
After editing the file, you add it to the staging area to indicate that the conflict has been resolved:

            git add index.html
Finally, you commit the changes to finalize the resolution and complete the merge:

            git commit -m "Resolved merge conflict by incorporating both header and title changes"
Now, the main branch includes both your and your teammate's contributions, and the project moves forward without any hitches.
